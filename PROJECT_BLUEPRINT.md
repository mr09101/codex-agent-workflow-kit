## 1. 제품 요구사항

- 상태: 구현·검증 완료
- 해결 문제: AI 코딩 에이전트 작업에 필요한 운영 문서 네 개를 생성하고 필수 섹션을 검사합니다.
- 주요 사용자: Codex 또는 유사 에이전트를 사용하는 공개 OSS 유지관리자입니다.
- 핵심 흐름: `init <target>`으로 템플릿을 만들고 `check <target>`으로 구조를 검사합니다.
- 성공 기준: 정상 생성과 검사가 통과하고, 기존 파일은 기본적으로 보존되며, 안전하지 않은 링크 경로는 거부됩니다.
- 근거: `README.md`, `codex_multi_agent_workflow_kit/cli.py`, `tests/test_cli.py`.

## 2. 현재 범위와 제외 범위

- 상태: 구현·검증 완료
- 현재 범위: 네 개의 Markdown 템플릿 생성, 필수 파일·제목 검사, 공개 가능한 CLI 결과와 종료 코드입니다.
- 제외 범위: AI 에이전트 실행, API 호출, 인증, 비밀값 관리, 웹 UI, 서버, 데이터베이스, 배포 자동화입니다.
- 이번 변경 범위: filesystem link/reparse 경계, 예상 가능한 경로·인코딩 오류, CI 최소 권한, 관련 문서와 테스트입니다.
- 범위 변경 승인 기준: 새 네트워크·영속 저장·UI 기능은 별도 위협 모델과 테스트 계획이 승인된 뒤 추가합니다.

## 3. 아키텍처와 데이터 모델

- 상태: 구현·검증 완료
- 스택: Python 3.10 이상 표준 라이브러리와 `unittest`.
- 구성요소: `cli.py`의 파서·생성·검사 로직, 인메모리 템플릿 사전, Markdown 문서, GitHub Actions입니다.
- 데이터 흐름: 명령 인자에서 대상 경로를 받아 고정 템플릿을 파일로 쓰거나 필수 제목을 읽어 검사합니다.
- 영속 데이터 모델: 해당 저장소가 관리하는 데이터베이스나 사용자 상태는 없습니다.
- 신뢰 경계: 사용자가 지정한 대상 디렉터리와 그 안의 기존 filesystem 객체는 신뢰하지 않습니다.
- 현재 단일 장애점: 로컬 filesystem 권한·상태에 의존하며 실패 시 비정상 종료 코드로 반환합니다.

## 4. 보안·개인정보 기준

- 상태: 구현·검증 완료
- 전체 위험도: HIGH 해결, MEDIUM 해결
- 보호 자산: 대상 루트 밖 파일, 로컬 경로 정보, 공개 저장소의 비밀값·개인정보입니다.
- 진입점: `init`/`check` 대상 경로, 대상 내부의 기존 파일·symlink·junction·reparse point, 검사 파일의 UTF-8 내용입니다.
- HIGH 해결: target 루트와 생성 경로의 link/reparse point를 검사하고 canonical target root 밖 해석을 거부하며 `--force`도 우회하지 못하게 했습니다.
- MEDIUM 해결: 예상 가능한 filesystem·UTF-8 예외를 traceback 대신 공개 가능한 한 줄 오류와 exit code `2`로 변환했습니다.
- 잔여 위험: 신뢰하지 않는 프로세스가 검사와 쓰기 사이에 filesystem을 동시에 바꾸는 경쟁 조건은 CLI sandbox 범위 밖입니다. 해당 디렉터리에서 동시 실행하지 않도록 `SECURITY.md`에 명시했습니다.
- 개인정보: 수집·전송·저장이 없으며 실제 키, 계정, 이메일, 개인 경로를 문서화하지 않습니다.
- 민감 파일 기준: `.env*`, 로그, 사용자 데이터, 로컬 에이전트 상태는 추적·스테이징하지 않습니다.

## 5. 트래픽·확장성 가정

- 상태: 해당 없음
- 이유: 현재 제품은 요청을 받는 서버가 아니라 한 사용자가 로컬에서 실행하는 단일 프로세스 CLI이므로 동시 서버 요청, 페이지네이션, 캐시, 큐, 수평 확장 지표가 없습니다.
- 현재 사용량 가정: 한 번에 대상 저장소 하나와 작은 고정 템플릿 네 개를 처리합니다. 이는 측정된 운영 트래픽이 아닌 제품 구조상의 가정입니다.
- 도입하지 않는 것: 데이터베이스 인덱스, 캐시, 비동기 작업 큐, 복제, 수평 확장입니다.
- 재검토 조건: 웹 UI/API 서비스화, 원격 실행, 다중 사용자 동시 작업, 사용자 정의 대량 템플릿 또는 큰 입력 파일 지원 시 수치 측정과 용량 기준을 새로 정의합니다.

## 6. 장애·운영 대응

- 상태: 구현·검증 완료
- 정상 종료: 생성·검사 성공은 exit code `0`입니다.
- 검증 실패: 필수 파일·섹션 누락은 exit code `1`입니다.
- 실행 오류: 안전하지 않은 경로, filesystem 실패, 잘못된 UTF-8은 한 줄 오류와 exit code `2`입니다.
- 운영 표면: Python 3.10~3.12 단위 테스트 CI와 Python 3.12 init/check smoke CI입니다.
- 외부 장애: 네트워크·외부 API·rate limit·재시도는 외부 호출이 없어 해당 없습니다. 외부 연동 추가 시 재검토합니다.
- 복구: 기본 no-overwrite로 기존 파일을 보존하며, 실패 원인을 해소한 뒤 같은 명령을 다시 실행합니다.

## 7. 테스트와 검수 기준

- 상태: 구현·검증 완료
- 단위 테스트: 정상 생성·검사, 누락 파일·섹션, 기본 no-overwrite, `--force`, symlink/junction/reparse 경계, 경로 충돌, 잘못된 UTF-8을 검사합니다.
- Windows 조건: 파일 symlink 권한이 없으면 권한 사유를 명시해 해당 사례를 skip하고, 권한이 필요 없는 junction으로 디렉터리와 target root 경계를 실제 검증합니다.
- 전체 검증 명령: 번들 Python으로 `python -m unittest discover -s tests -v`.
- 정적 검증: 번들 Python으로 패키지와 테스트를 `compileall`.
- smoke: 임시 대상에 `init` 후 `check`, 기존 파일 no-overwrite, junction `--force` 차단을 확인합니다.
- UI 검수: 해당 없음. 실제 웹 UI가 생기면 모바일·데스크톱 렌더링 검수를 추가합니다.

## 8. 향후 확장 경로

- 상태: 계획됨
- 다음 단계: 기존 저장소의 로컬 규칙을 덮어쓰지 않고 도입하는 지침을 추가합니다.
- 후속 단계: 필요가 확인될 때 템플릿 사용자 정의와 검사 규칙 확장을 별도 설계합니다.
- 현재 보류: 에이전트 런타임, 외부 API, 데이터베이스, 웹 서비스, 분산 실행은 제품 핵심을 벗어나므로 도입하지 않습니다.
- 재설계 조건: 사용자 요구가 고정 템플릿 네 개와 로컬 단일 저장소 모델을 넘을 때 인터페이스와 위협 모델을 먼저 검토합니다.

- 수정 대상: CLI, 단위 테스트, 두 GitHub Actions 워크플로, README, SECURITY, HANDOFF, PROJECT_BLUEPRINT입니다.
